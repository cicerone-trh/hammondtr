Programming

First is the nebulous concept of programming. There are two aspects I can divide my passion into: the abstract, romantic notion of the undefinable limits of computers and the tangible, practical satisfaction of crafting something to accomplish a goal.

There is nothing special about a Turing machine in itself. Computers are remarkable because of the human ability to abstract meaning. We set systems up to represent any process we can abstract and a computer can embody that. Further, the only things that 'exist' are relative abstractions that our mind creates. Because of this one to one relationship (reality abstraction : computer abstraction), the only limit to what can be done with a computer is the limits of our perception. Or, more eloquently, anything is possible with a machine that can process lists.

Another way to put it is: everything we know about the world is a model. Computers can embody any model we imagine. Thus, computers are capable of anything.

But that's just the romantic part of programming that only rarely makes you swoon; that realization is a clear night, with a new moon in the country and the stars are so vivid you notice the Orion Nebula for the first time.

The true romance is the practical nature of the beast: it is an opportunity to build something completely new out of nothing, to really test one's mental craftsmanship. Indeed, the sort of mental acrobatics required and the satisfaction of having built something that provides tangible results are what make the discipline so enticing. It is a modern-day artisanal profession: the languages and frameworks are the medium from which to shape a functional experience.

It's the satisfaction of getting something working just like you wanted it to, when the pieces all come together and you just want to play with it.

Video Games

I like video games of all types, but the ones I'm most absorbed with are competitive. If Thomas Was Alone is the occassional dessert, Dota 2 is a basic meal.

Dota 2 is a complex system that professionals spend thousands of hours understanding. Because of its complexity, there is an astounding difference in understanding between me and a professional. Thus, almost every game there's something new to learn and something to be awed by.

Like a mental version of parkour: the simply stated goal is to get from point a to b as efficiently as possible, but the system of their movement is remarkable; it's revolutionary compared to your current understanding of how to get from point a to point b. It's inspiring to always be in awe of the capabilities of humanity.

While the most stimulating part is the macro strategy, the micro can be pretty spectacular too. This clip still gives me goosebumps ^_^

Driving

Driving is perhaps the least suited of these passions to express via text. You just have to live it and understand it; there's nothing quite like going fast. It's thrilling and it's incredibly easy to get into flow identifying the limit of your vehicle as you take a turn.

But that's only one variant of driving. Another variant of driving would be dirt bikes. This is even harder to express via text, because far fewer people have the experience to build on.

When you drive a car, there's subconsciously a room around you: you are closed off from your environment. When you're on a motorcycle, you're almost completely exposed to the world. Because of this, you get a real sense of "going" somewhere. I tried to explain this to a friend by saying, "cars are like teleporting but very slow, while bikes are like walking but very fast". Also, you have all the potential thrill from driving a car and melding with the machine.

Combine this with outdoors exploring, and you have an incredibly rich experience.

Juggle Fest ::

  • Program.cs
  • FestivalCoordinator.cs
  • Juggler.cs
  • Circuit.cs

This is a solution to a programming challenge* offered by Yodle. In short, the challenge asks you to write a program that sorts out a large number of Jugglers based on various attributes and desires such that the best teams perform on their most desired Circuits. Or, you can see how I rephrased it in the comments: (FestivalCoordinator, Line 53)

As previously stated, these examples are of limited scope. If this program used polymorphic behaviors on inherited classes it would be overly complex. Example, the Coordinator could operate on a "person" and "juggler" could be a type of person, but that's pointlessly complex (and definitely not agile).

That said, because of proper encapsulation and sensible object divisions, it would be fairly easy to change this functionality. Maybe the Coordinator has to coordinate "Jugglers" and "Bowlers", in that case it might be better to have an "Athlete" class; you wouldn't need to change the function names from Juggler, and thus the Coordinator wouldn't have to change very much.

Next, the key to solving the sorting problem is recursion: when a team is full, but you come across another Juggler that is better suited, you have to shuffle that displaced Juggler onto another team. That Juggler can displace another and so on. Thus, a recursive function is the solution: (FestivalCoordinator, line 112)

Still, I have included this section because I found assembly to be extremely enlightening. I think a stint in low-level C or asm is somewhat of a pilgrimage all programmers should take. hammondtr.net - github

This site is 100% front-end and all running code can be found through view source. It is animated with a combination of jQuery and CSS3. While I understand people generally prefer to scroll rather than to click, I personally enjoy clicking buttons. If you prefer a scrolling site, check back in like.. a year. A redesign that's scrolly and fully responsive is on my list of projects.

My favorite part is the code window and its navigation. It is implemented with an iframe and html files generated with VIM's :tohtml function. In order to manipulate the iframe I've written a function "setSourceWindow" that takes an iframe element, a uri, and an optional integer as arguments.

The function creates a jQuery selector for the line-number, loads the resource if needed, and then uses a jQuery function to scroll the page to the selector. It's simple, but I really like the effect.

Ambler Architects - staging.amblerarchitects.com

This site is what I'd call a 'regular site', I wrote the front-end and implemented the PyroCMS back-end. It has multiple pages of content, including a portfolio section, and it uses a CMS to allow non-technical users to edit this content.

I have just checked this site and it is not displaying properly on Linux/MacOS browsers. The issue is that the CSS framework being used addresses whitespace between “stacked” inline-block elements by applying a negative margin equal to the width of the parent font's whitespace. The source of the original font has expired, and it has now fallen back to browser default, which on certain machines has a 1px wider whitespace. This could have been avoided by specifying an alternate font, but when implementing the site, I was told to use only the one selected.
